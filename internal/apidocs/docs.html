This page is intentionally boring. It documents what the server actually serves today, not what the roadmap might serve later.

  • Canonical status: GET /api/v1/status is the preferred machine-readable status route.
  • Canonical REST surface: new clients should target the documented /api/v1/... routes; older non-v1 routes are compatibility aliases or legacy helpers.
  • Fallbacks matter: some status fields are protocol-native, some are display-derived fallback, some are compatibility aliases, and some are omitted when unknown.
  • Alarms endpoint: /api/v1/alarms is still an explicit stub in phase 1.
  • Raw artifact: /api/v1/openapi.json

Status honesty notes

  • recentContact is a conservative recent-heard-from-the-amp indicator. It is not the same as TX or operate state.
  • bandCode, warningCode, and alarmCode are protocol-backed raw fields when present.
  • bandText, warningsText, and alarmsText are decoded companions, not stronger truth than the raw codes.
  • warnings and activeAlarms are compatibility aliases retained for consumers already using them.
  • /api/v1/telemetry is still useful, but it should not be treated as more authoritative than /api/v1/status.
  • Older routes like /api/status and /api/telemetry still exist for compatibility, but this docs UI describes the canonical /api/v1/... surface.
